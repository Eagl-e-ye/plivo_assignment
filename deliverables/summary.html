End-of-Turn Detection using Prosodic Features and CNN Ensemble

This report describes the complete approach followed for solving the End-of-Turn (EOT) Detection challenge. The objective was to estimate the probability that every pause in a spoken utterance represents the actual end of the speaker's turn while satisfying the strict causality constraint that no future audio after the pause can be used.

1. Problem Understanding

The task was not simply silence detection. Many pauses occur naturally while people think, breathe or search for words. A voice assistant that replies immediately after every pause will interrupt the speaker, while waiting too long creates an unnatural delay. Therefore, the problem becomes a binary classification task:

  • Hold : speaker will continue speaking.
  • End-of-Turn : conversation should be handed over to the AI.

The challenge was to make this decision using only information available before the pause begins.

2. Initial Thought Process

The first idea considered was using silence duration as the only feature. Although longer pauses generally indicate the end of a sentence, after listening to multiple recordings it became clear that speakers often paused for similar durations before continuing. Therefore silence alone was insufficient.

Instead of relying on only one signal, the idea evolved into combining multiple cues that humans naturally use while listening:

  • How the pitch changes before stopping.
  • How loud the speaker is before the pause.
  • Whether the speaker is slowing down.
  • The speaking rate near the pause.
  • The spectral characteristics of the last spoken segment.

These observations motivated building a hybrid model capable of combining handcrafted speech features with automatically learned representations.

3. Overall Approach

The final solution consists of two independent models whose outputs are combined using weighted averaging.

Model Purpose
Extra Trees Classifier Uses handcrafted prosodic and statistical features.
CNN Learns patterns directly from MFCC representations.
Weighted Ensemble Combines strengths of both models for final probability.

Final prediction:

P(EOT) = w₁ × P(Tree) + w₂ × P(CNN)

4. Feature Engineering

Only the speech preceding every pause was analysed. No information after the pause was used, ensuring compliance with the causality requirement.

The handcrafted feature vector included:

  • RMS energy
  • Pitch statistics
  • Pitch slope
  • Energy trend
  • Zero Crossing Rate
  • Spectral Centroid
  • Spectral Bandwidth
  • Spectral Roll-off
  • MFCC statistics
  • Speech duration before pause

These features capture changes in intonation and speaking behaviour near the end of an utterance.

5. Input Representation

The two models operate on different representations of the same speech segment extracted from the last 1.5 seconds before every pause.

Model Input Representation Input Dimension
Extra Trees Handcrafted acoustic and prosodic feature vector (1 × N), where N is the total number of extracted features.
CNN MFCC Feature Map (1 × 40 × 150)

For the Extra Trees classifier, each pause is represented as a single one-dimensional feature vector containing statistical and prosodic information extracted from the preceding speech segment.

For the CNN, the same speech segment is converted into a Mel Frequency Cepstral Coefficient (MFCC) representation with 40 coefficients. The temporal dimension is either padded or truncated to 150 frames, producing an input tensor of shape (1, 40, 150). During inference, PyTorch adds the batch dimension, resulting in an input tensor of (1, 1, 40, 150), where:

  • First dimension → Batch size
  • Second dimension → Single MFCC channel
  • Third dimension → 40 MFCC coefficients
  • Fourth dimension → 150 time frames

6. CNN Branch

Although handcrafted features capture interpretable speech characteristics, they may fail to model complex spectral patterns. To overcome this limitation, a CNN was trained using MFCC feature maps.

Pipeline:

  1. Extract last 1.5 seconds of speech before pause.
  2. Generate MFCC representation.
  3. Normalize features.
  4. Feed into convolutional neural network.
  5. Predict probability of End-of-Turn.

7. Why Ensemble Learning?

During experimentation it was observed that handcrafted features performed well on structured speech while the CNN generalized better to difficult examples. Instead of selecting one model, both predictions were combined using weighted averaging. This reduced variance while retaining the advantages of both approaches.

8. Development Process

The project was developed iteratively instead of building the final model in a single step. The overall workflow followed was:

  1. Understand dataset structure.
  2. Study pause annotations.
  3. Listen to sample recordings.
  4. Implement silence baseline.
  5. Engineer handcrafted speech features.
  6. Train traditional ML classifiers.
  7. Evaluate performance.
  8. Introduce CNN using MFCC inputs.
  9. Compare both models.
  10. Build weighted ensemble.
  11. Tune ensemble weights.
  12. Generate predictions.

15. Conclusion

The final solution combines handcrafted speech analysis with deep learning through an ensemble framework. Instead of depending solely on pause duration, the model attempts to infer speaker intent from multiple acoustic cues observed before every pause. The overall pipeline follows the competition constraints and is designed to generalize to unseen conversations while balancing response speed and false interruption rate.
